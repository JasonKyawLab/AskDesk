# Deploying AskDesk

Same code, two shapes вЂ” chosen by whether `ASKDESK_REDIS_URL` is set:

- **All-in-one** (no Redis) вЂ” one process runs everything. Best for the free
  tier. в†’ Part A.
- **Web + worker** (Redis set) вЂ” a thin web tier enqueues; a separate `worker`
  runs the engine. Best for a paid, always-on host. в†’ Part B.

The bot must be reachable over public HTTPS (hosted or tunneled) before Telegram
can deliver messages.

### Pick your channels вЂ” Telegram, Web API, or both

You don't have to run both channels. They're independent, driven by config:

| You want | Set | Result |
|---|---|---|
| **Telegram only** | `ASKDESK_TELEGRAM_BOT_TOKEN` (skip the Web API) | the bot; the Web API simply goes unused |
| **Web app only** | leave the bot token empty | just the `/api/v1` endpoints вЂ” no Telegram at all |
| **Both** | set the bot token | both, sharing one database and FAQ set |

The Web API is available whenever a database is configured; Telegram turns on
only when a bot token is set. Everything below applies to whichever you pick вЂ”
web-only setups skip the Telegram steps and manage FAQs, settings, and the
pending-question queue through `make admin-link` instead (see the Web API
section).

## Part A вЂ” Free: Render (web) + Supabase (Postgres)

Render's free web service is free forever (it sleeps after 15 min idle).
Postgres comes from **Supabase** (free forever, with pgvector) вЂ” not Render's
Postgres, whose free tier expires after ~30 days.

### 1. Postgres on Supabase
1. Create a project (set a DB password вЂ” letters and numbers only, no symbols).
2. Database в†’ Extensions в†’ enable **`vector`**.
3. Click **Connect** (top bar) в†’ **Session pooler** в†’ Type: **URI** в†’ copy it.
   - вљ пёЏ Use the **Session pooler** (`...pooler.supabase.com:5432`), **not** the
     Transaction pooler (6543) and **not** Direct connection (IPv6-only вЂ” Render
     is IPv4). Migrations need a session-level advisory lock the transaction
     pooler can't hold.
   - Replace `[YOUR-PASSWORD]` and append `?sslmode=require`. This is your
     `ASKDESK_DATABASE_URL`.

### 2. Bot + AI keys
- Create a bot with **@BotFather** в†’ copy the bot token.
- Your Telegram numeric id (you become the admin): message **@myidbot** в†’ `/getid`.
- Get a **Gemini API key** at ai.google.dev.

### 3. Deploy on Render
Use the included `render.yaml` (New в†’ Blueprint в†’ pick the repo), or create a
**Web Service в†’ Docker** manually. Environment variables:

| Key | Value |
|---|---|
| `ASKDESK_DATABASE_URL` | Supabase **Session-pooler** URI (`sslmode=require`) |
| `ASKDESK_GEMINI_API_KEY` | your Gemini key |
| `ASKDESK_TELEGRAM_BOT_TOKEN` | your bot token |
| `ASKDESK_BUSINESS_ID` | `1` |
| `ASKDESK_TELEGRAM_WEBHOOK_SECRET` | auto-generated by Render (note its value) |
| `ASKDESK_MAGIC_LINK_SECRET` | auto-generated by Render |

Leave `ASKDESK_REDIS_URL` **unset** (в†’ all-in-one). Don't set `PORT` or
`ASKDESK_PUBLIC_URL` вЂ” the app uses Render's `PORT` and `RENDER_EXTERNAL_URL`.

Deploy. On first boot the app connects to Supabase and **runs migrations
automatically** (they're embedded in the binary). Check `/healthz` в†’ `{"status":"ok"}`.

### 4. Seed your business + admin (one time)
In Supabase в†’ SQL Editor (put YOUR Telegram id):

```sql
INSERT INTO businesses (id, name, api_key)
  OVERRIDING SYSTEM VALUE VALUES (1, 'Your Shop', gen_random_uuid())
  ON CONFLICT (id) DO NOTHING;
INSERT INTO admins (business_id, channel, external_id, name)
  VALUES (1, 'telegram', 'YOUR_TELEGRAM_ID', 'owner')
  ON CONFLICT DO NOTHING;
```

(or `make seed BUSINESS_NAME="Your Shop" ADMIN_TG_ID=<id>` with `ASKDESK_DATABASE_URL` exported).

### 5. Connect the bot
```
export ASKDESK_TELEGRAM_BOT_TOKEN="<bot token>"
export ASKDESK_PUBLIC_URL="https://<your-app>.onrender.com"
export ASKDESK_TELEGRAM_WEBHOOK_SECRET="<the value Render generated>"
make set-webhook
```

### 6. Load FAQs + brand it
- **FAQs:** write `faqs.json` (`[{category, question, answer}, ...]`), then
  `make load-faqs file=faqs.json` (needs `ASKDESK_DATABASE_URL` +
  `ASKDESK_GEMINI_API_KEY`). Update later with `... reset=1`.
- **Branding:** message the bot `/admin` в†’ **вњЏпёЏ Edit FAQs** в†’ set shop name,
  welcome, fallback, and ask-prompt (use `{name}` for the shop name). Effective
  on the next message вЂ” no redeploy.

### 7. Use it
- **Customers:** share `https://t.me/<yourbotusername>`. They tap Start в†’ menu в†’
  a topic for an instant answer, or type a question for the AI.
- **You:** `/admin` в†’ рџ“Љ stats В· рџ“Ґ pending questions (tap в†’ вњЌпёЏ Reply or рџ—‘ Dismiss)
  В· вњЏпёЏ Edit FAQs.

**Keep it awake (optional):** a free cron (cron-job.org) pinging
`https://<your-app>.onrender.com/wake` every 5вЂ“10 min stops Render sleeping.

## Part B вЂ” Paid / always-on: Oracle Cloud (OCI)

On a box you control you can run all-in-one (like the free tier, just on a bigger
always-on server) **or** the web + worker split for independent scaling.

1. **Server:** an OCI VM with Docker + docker compose. Open 443.
2. **Data:** run Postgres (pgvector) + Redis (the bundled `docker-compose.yml`
   does this for dev; use managed/hardened instances in production, never public).
3. **Environment:** same as Part A, **plus** `ASKDESK_REDIS_URL=redis://...` and
   `ASKDESK_PUBLIC_URL=https://your-domain`. Setting Redis switches on the split.
4. **Run both tiers** from the one image (`make docker-build`):
   - web: entrypoint `/askdesk` (default)
   - worker: entrypoint `/worker`
   Scale the worker by running more replicas.
5. **HTTPS:** put Caddy or nginx in front for TLS, then `make set-webhook` and
   seed as in Part A.

**Same code both ways** вЂ” moving from free to paid is env vars + (optionally)
running the `worker`, not a code change.

## Web API (use it from your own web app)

The JSON API is available on any deployment (no Telegram required). Base URL
`https://<your-app>/api/v1`, header `X-API-Key: <business api_key>` (find it with
`SELECT api_key FROM businesses WHERE id = 1;`).

| Endpoint | |
|---|---|
| `GET /api/v1/config` | shop name, welcome, ask prompt, categories |
| `GET /api/v1/faqs` | categories with `{id, question, answer}` |
| `POST /api/v1/ask` | body `{message, session_id}` в†’ `{answer, answered}` |
| `GET /api/v1/replies?session_id=&since=` | admin replies for that session (poll it) |

Pass a stable random `session_id` from your frontend (or omit it в†’ `anon`). It's
just a grouping key вЂ” no personal data required.

CORS: set `ASKDESK_CORS_ORIGINS` to your site (or `*`). Only needed for direct
browser calls.

### Answering web customers (handoff)
When the AI can't answer a web question, it's queued like any other. Reply to it
from **any** admin surface вЂ” all route the answer back to the web visitor, who
receives it by polling `/api/v1/replies`:

- **Telegram** (if set up): `/admin` в†’ рџ“Ґ Pending в†’ tap в†’ вњЌпёЏ Reply.
- **Built-in web page** (no Telegram): run `make admin-link` (needs
  `ASKDESK_MAGIC_LINK_SECRET` + `ASKDESK_PUBLIC_URL`) в†’ open the link в†’ pending
  questions with a reply box, plus FAQ/settings editing.
- **Your own admin UI** (build it into your app): the admin JSON API below.

### Admin API вЂ” build your own support inbox

For answering from inside your own app, call the privileged admin API. It uses a
**separate** `X-Admin-Key` header (find it with `SELECT admin_api_key FROM
businesses WHERE id = 1;`) and sends **no CORS headers** вЂ” call it **from your
backend**, never a browser (the admin key must stay server-side).

| Endpoint | |
|---|---|
| `GET /api/v1/admin/stats` | `{total, answered, unanswered}` |
| `GET /api/v1/admin/pending` | `{pending: [{id, question, customer}]}` |
| `POST /api/v1/admin/reply` | body `{id, message}` в†’ delivers to the customer's channel, resolves |
| `POST /api/v1/admin/dismiss` | body `{id}` в†’ resolve without replying |

The public `api_key` is rejected here вЂ” only the `admin_api_key` works. Rotate
either key with `UPDATE businesses SET admin_api_key = gen_random_uuid()::text
WHERE id = 1;`.

**Security:** the API is read-only and tenant-isolated вЂ” a key only reads its own
business's public FAQs, and nothing can be written through it. The one caveat:
`POST /ask` spends your Gemini quota and is **not yet rate-limited**, so call it
from your **backend** (key stays server-side), or lock CORS to your domain and
rate-limit `/ask` yourself.

## Switching modes

Nothing in the code changes. Add `ASKDESK_REDIS_URL` (+ run a `worker`) for the
split; remove it for all-in-one. Same image, same binaries.
