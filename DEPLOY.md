# Deploying AskDesk

The same code runs two ways, chosen by whether `ASKDESK_REDIS_URL` is set:

- **All-in-one** (no Redis) — one process; the webhook runs the engine inline.
  Best for the free tier. → Part A.
- **Web + worker** (Redis set) — a thin web tier enqueues; a separate worker
  runs the engine. Best for a paid always-on host. → Part B.

Telegram must be able to reach your server over public HTTPS, so the app has to
be hosted (or tunneled) before the bot works.

---

## Part A — Free forever: Render (web) + Supabase (Postgres)

Render's free **web service** is free forever (it sleeps after 15 min idle).
Postgres comes from **Supabase** (free forever, with pgvector) — *not* Render's
Postgres, whose free tier expires after ~30 days.

### 1. Postgres on Supabase
1. Create a Supabase project.
2. Make sure the `vector` extension is available (Database → Extensions →
   enable `vector`). The app's migration also runs `CREATE EXTENSION IF NOT
   EXISTS vector`, but enabling it here avoids permission surprises.
3. Copy the **direct** connection string (Project Settings → Database →
   Connection string → **URI**, port **5432**).
   - ⚠️ Use the **direct/session** connection (port 5432), **not** the
     transaction pooler (port 6543). Migrations take a session advisory lock,
     which the transaction pooler does not support — via the pooler, startup
     hangs. Append `?sslmode=require`.

### 2. Bot + AI keys
- Create a bot with **@BotFather** → copy the **bot token**.
- Get a **Gemini API key** at ai.google.dev.

### 3. Deploy on Render
Either use the included `render.yaml` (New → Blueprint → pick this repo), or
create a **Web Service** → **Docker** from the repo manually. Then set env vars:

| Key | Value |
|---|---|
| `ASKDESK_DATABASE_URL` | Supabase **direct** URI (port 5432, `sslmode=require`) |
| `ASKDESK_GEMINI_API_KEY` | your Gemini key |
| `ASKDESK_TELEGRAM_BOT_TOKEN` | your bot token |
| `ASKDESK_BUSINESS_ID` | `1` |
| `ASKDESK_TELEGRAM_WEBHOOK_SECRET` | auto-generated by Render (note it down) |
| `ASKDESK_MAGIC_LINK_SECRET` | auto-generated by Render |

You do **not** set `ASKDESK_PUBLIC_URL` or `ASKDESK_HTTP_PORT` — the app falls
back to Render's injected `RENDER_EXTERNAL_URL` and `PORT`.
Leave `ASKDESK_REDIS_URL` **unset** → all-in-one mode.

Deploy. On first boot the app connects to Supabase and **runs the migrations
automatically** (schema is embedded in the binary). Check `/healthz` returns ok.

### 4. Seed the first business + admin
The first admin is added directly in the DB (identity-based auth — no passwords).
Find your Telegram numeric id by messaging **@userinfobot**. Then, locally:

```
export ASKDESK_DATABASE_URL="<supabase direct URI>"
make seed BUSINESS_NAME=minipos ADMIN_TG_ID=<your-telegram-id>
```

(or run the two INSERTs from the Makefile in Supabase's SQL editor.)

### 5. Point Telegram at your service
```
export ASKDESK_TELEGRAM_BOT_TOKEN="<bot token>"
export ASKDESK_PUBLIC_URL="https://<your-service>.onrender.com"
export ASKDESK_TELEGRAM_WEBHOOK_SECRET="<the value Render generated>"
make set-webhook
```

### 6. Use it
- Message your bot `/edit` → tap the link → add FAQs.
- Ask the bot a question → it answers from your FAQs.
- `/stats`, `/unanswered` for admin.

**Note on sleep:** a free Render service sleeps after 15 min idle, so the first
message after a quiet period takes ~30–60s to wake it (Telegram retries, so it
still arrives). Optional: keep it warm with a free cron ping to `/wake` every
10 minutes (e.g. cron-job.org). `/wake` touches no dependencies, so it responds
instantly and just wakes the instance.

## Part B — Paid / always-on: Oracle Cloud (OCI) with the worker split

On a box you control (e.g. OCI Ampere free VM, or any paid instance) you can run
the full split: web + worker + Redis + Postgres.

### 1. On the server
Install Docker + docker compose. Run Postgres (pgvector) and Redis — the bundled
`docker-compose.yml` does this for local dev; in production use managed services
or hardened containers, and never expose their ports publicly.

### 2. Environment
Set the same variables as Part A, **plus**:

| Key | Value |
|---|---|
| `ASKDESK_REDIS_URL` | `redis://<redis-host>:6379/0` |
| `ASKDESK_PUBLIC_URL` | your public HTTPS URL |

Setting `ASKDESK_REDIS_URL` switches the web binary into thin-web (enqueue) mode.

### 3. Run both tiers
Build the image (`make docker-build`) and run two containers from it:
- **web:** entrypoint `/askdesk` (default) — receives webhooks, enqueues.
- **worker:** entrypoint `/worker` — consumes the queue, runs the engine.

Scale the worker independently by running more replicas.

### 4. HTTPS + webhook
Put a reverse proxy (Caddy or nginx) in front for TLS (Let's Encrypt), then
`make set-webhook` as in Part A. Seed the business/admin the same way.

---

## Switching between modes

Nothing in the code changes. Add `ASKDESK_REDIS_URL` (+ run a `worker`) to move
from all-in-one to the split; remove it to go back. Same image, same binaries.
